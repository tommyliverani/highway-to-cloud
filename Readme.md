### COME SIAMO ARRIVATI QUI

Una volta si lavorava a reparti stagni(classica immagina dei silos)
Con il Devops si tenta di abbattere il muro tra dev e ops -> you build it, you run it
Obiettivi del devops(Dora metrics - Accelerate: The Science of Lean Software and DevOps - 2018):
- Deploymnent frequency
- Leat time for changes
- Change failure rate
- Time to restore service

Con una platform si tenta di abbattere il carico cognitivo introdotto dalla filosofica devops. "you build it, the platform will run it for you".

Obiettivi classici di una platform:

- Riduzione del carico cognitivo
- Abilitazione del sef-service
- standardizzazione e secuirty by design


### DIFFICOLTÀ NELL'AFFRONTARE UN PROGETTO PER LA COSTRUZIONE DI UNA IDP

La platform non crea valore direttamente(non creo un nuovo servizio, a meno che la platform stessa non sia il mio prodotto) ma indirettamente(abilita/facilità chi crea valore).
Spesso le stesse persone che dovrebbero ricevere il valore della platform sono le stesse che generano frizzioni nell'adozione della stessa(cambia il modo di lavorare).
I tool e i paradigmi di un contesto devops cambiano molto velocemente.
I servizi forniti dai cloud provider aumentano di anno in anno.
Le capability che la platform deve abilitare possono essere numerose e facilmente modificabili
Per implementare una richiesta spesso sono necessarie molteplici competenze e autorizzazioni.

-> Necessità di una platform facilmente evolvibile e senza locking tecnologici in grado di dare valore nel tempo


### NUOVI OBIETTIVI PER FARE UNA PLATFORM DI SUCCESSO

- Riduzione del carico cognitivo
- Abilitazione del sef-service
- Standardizzazione e security by design
- Indipendente dalla tecnologia 
- Rapida implementazione di nuove capability
- Responsabilità ben definite
